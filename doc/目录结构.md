# 我建议的核心原则

# 1. 数据源独立

这是最重要的。

你提到：

> 官方源和社区源，每一个源建立一个文件夹

这个方向是对的。

因为：

每个平台：

* 抓取方式不同
* 反爬不同
* 数据结构不同
* 清洗逻辑不同

后面一定会分化。

---

# 2. “采集” 和 “AI处理” 必须隔离

很多人会写成：

```text
抓取 → 直接AI总结
```

后面必死。

因为：

以后你一定会：

* 重跑摘要
* 更换模型
* 重跑标签
* 做embedding
* 做趋势分析

所以：

# 原始数据必须保留

# 这里面最关键的设计（重点）

# 一、每个源单独目录

例如：

```text
sources/official/openai/
```

里面：

```text
crawler.py
parser.py
cleaner.py
```

因为：

# 每个平台后面一定会越来越特殊

例如：

| 平台         | 特殊性       |
| ---------- | --------- |
| Reddit     | 热度算法      |
| Twitter    | 线程解析      |
| GitHub     | Release解析 |
| OpenAI     | Blog结构    |
| HackerNews | 评论排序      |

独立目录非常正确。

---

# 二、storage/raw 非常重要

这是很多人会忽略的。

你必须保留：

# 原始文章

因为以后你可能：

* 更换 Prompt
* 更换模型
* 重做摘要
* 重新分类
* 做向量搜索

如果原文丢了：
系统直接废掉。

---

# 三、pipelines 独立（非常关键）

很多人后面会崩。

因为逻辑全部写在 crawler。

正确做法：

```text
source只负责获取
pipeline负责处理
```

例如：

```text
openai/crawler.py
```

只负责：

```python
获取原始文章
```

而：

```text
summarize_pipeline.py
```

才负责：

```python
AI总结
```

这个边界非常重要。

---

# 四、AI层独立（重点）

因为：

你后面一定会：

* OpenAI
* Claude
* Gemini
* 本地模型

混用。

所以：

# 不要在业务代码里直接调用OpenAI

必须：

```text
ai/llm/router.py
```

统一调度。

---

# 五、为什么 publishing 要独立

因为：

内容生产 和 内容分发 是两回事。

以后你会：

* Telegram
* 微信
* 邮件
* Web
* API

统一从：

```text
digest_service
```

输出。
